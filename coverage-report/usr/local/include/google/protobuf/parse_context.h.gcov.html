      73             : // The basic abstraction the parser is designed for is a slight modification
      74             : // of the ZeroCopyInputStream (ZCIS) abstraction. A ZCIS presents a serialized
      75             : // stream as a series of buffers that concatenate to the full stream.
      76             : // Pictorially a ZCIS presents a stream in chunks like so
      77             : // [---------------------------------------------------------------]
      78             : // [---------------------] chunk 1
      79             : //                      [----------------------------] chunk 2
      80             : //                                          chunk 3 [--------------]
      81             : //
      82             : // Where the '-' represent the bytes which are vertically lined up with the
      83             : // bytes of the stream. The proto parser requires its input to be presented
      84             : // similarly with the extra
      85             : // property that each chunk has kSlopBytes past its end that overlaps with the
      86             : // first kSlopBytes of the next chunk, or if there is no next chunk at least its
      87             : // still valid to read those bytes. Again, pictorially, we now have
      88             : //
      89             : // [---------------------------------------------------------------]
      90             : // [-------------------....] chunk 1
      91             : //                    [------------------------....] chunk 2
      92             : //                                    chunk 3 [------------------..**]
      93             : //                                                      chunk 4 [--****]
      94             : // Here '-' mean the bytes of the stream or chunk and '.' means bytes past the
      95             : // chunk that match up with the start of the next chunk. Above each chunk has
      96             : // 4 '.' after the chunk. In the case these 'overflow' bytes represents bytes
      97             : // past the stream, indicated by '*' above, their values are unspecified. It is
      98             : // still legal to read them (ie. should not segfault). Reading past the
      99             : // end should be detected by the user and indicated as an error.
     100             : //
     101             : // The reason for this, admittedly, unconventional invariant is to ruthlessly
     102             : // optimize the protobuf parser. Having an overlap helps in two important ways.
     103             : // Firstly it alleviates having to performing bounds checks if a piece of code
     104             : // is guaranteed to not read more than kSlopBytes. Secondly, and more
     105             : // importantly, the protobuf wireformat is such that reading a key/value pair is
     106             : // always less than 16 bytes. This removes the need to change to next buffer in
     107             : // the middle of reading primitive values. Hence there is no need to store and
